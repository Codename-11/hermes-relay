# Hermes-Relay Android — End-to-End UX & Daily-Use Audit (v1.0.0)

> Scope: first install → connect/pair → standard vs relay paths → every surface (Chat,
> Voice, Terminal, Manage, Bridge, Notifications, Settings) → Hermes management →
> daily-use friction ("non-technical daily user" test). Benchmarked against the Hermex
> iOS client and the Hermes desktop dashboard. Branch: `dev` @ 21938a6 + in-flight UI edits.
> Method: 4 parallel surface deep-dives + reference benchmark, synthesized.

---

## 1. Executive summary

Hermes-Relay is a **genuinely sophisticated, lovingly-built client** — the engineering
under the hood (streaming de-jitter, stale-while-revalidate Manage caching, per-host cookie
auth, fail-closed safety rails, multi-endpoint failover) is well beyond what most
self-hosted AI companions ship. The problems are **not** capability gaps; they're the gap
between *"thorough"* and *"effortless to use daily."*

**The central strategic tension.** The closest competitor, **Hermex**, sells radical
simplicity ("three moves, ~15 minutes," "no relay or third-party service," consumer polish)
but has **no voice, no terminal, no device control**. Hermes-Relay is the exact inverse:
it *wins decisively on power and breadth* but pays for it with a heavier setup story and a
**"relay" mental model Hermex never has to explain**. Almost every high-severity finding
below is a symptom of that tension leaking into the user's face.

**The one-sentence verdict:** *Make the Standard path feel as effortless as Hermex, so the
Relay power tools read as a bonus, not a tax.*

**Three cross-cutting themes drive ~70% of the friction:**

1. **Silent success/failure.** Edit-and-resend, slash commands, cancel/stop, voice
   unavailability, pairing timeouts, and revoked sessions all resolve into a transient
   bubble/toast or *nothing*. The app is usually *correct* in these paths but doesn't *tell
   the user it worked* — and a daily client lives or dies on that feedback loop.
2. **Mental-model overload / jargon.** "Relay," "pairing," "dashboard session," "API key,"
   "grants," "keyguard," "SOUL," "toolset," "MCP," "foreground service," "gateway/sessions/
   completions." A non-technical user is asked to hold three distinct auth concepts and a
   dozen protocol nouns before they can confidently use the thing.
3. **Standard ↔ Relay clarity.** What Standard gives vs. what Relay adds — and *why you'd
   pair the relay* — is implied by ordering but never made concrete, and the gates that
   guard relay-only features can dead-loop a Standard-only user.

**Top 5 highest-leverage fixes** (detail in §8):

| # | Fix | Why it matters | Effort |
|---|-----|----------------|--------|
| 1 | Add a distinct **"Pairing expired → Pair again"** state | Highest-frequency real failure — relay restart wipes sessions *every time*; today it looks like an infinite hang | M |
| 2 | Make **Verify/timeout errors context-aware** | Turns 3 dead-ends (API 401, host hasn't registered code, relay down) into self-explaining, recoverable errors | M |
| 3 | Tell the gate **"needs the Relay plugin on your server"** | Stops the Standard-only user's "Pair to unlock → nothing to pair to" dead loop | S |
| 4 | Tighten the **feedback loop** (haptics + transient "Ran /x ✓" / "Stopped" / "Still working…" states) | Kills the pervasive "did that work?" ambiguity across Chat/Voice | M |
| 5 | Kill **"Standard/Advanced" tier labels** on the Welcome screen | Violates the project's own design rule; reintroduces the tier confusion the redesign removed | S |

## 3. The first-run → working-connection journey

### 3.1 What actually happens (Standard path, the 95% case)

1. **Splash** held until DataStore + onboarding state load (`MainActivity.kt:72`).
2. **Onboarding pager** — Welcome → Chat → Manage → Power tools → **Connect**
   (`OnboardingScreen.kt:65`). Four info pages of value-prop before any input; "standard-first"
   ordering is correct (relay framed as optional "Power tools").
3. **Connect page** embeds the shared `ConnectionWizard` (`OnboardingScreen.kt:511`) — the same
   wizard used in Settings → Connections, so onboarding can't leave you "half-paired."
4. **Method step:** primary **Standard Hermes** tile, **Scan setup QR**, and an *"Advanced:
   Relay pairing"* section (`ConnectionWizard.kt:953`).
5. **Standard entry:** URL/host field that accepts a bare `192.168.1.10`, a one-tap **Scan for
   Hermes on LAN**, optional API key + Tailscale URL, Advanced URLs/ports expander.
6. **Readiness result card** — six per-surface status lines (Chat / Manage / Voice / Remote /
   Relay) with green/neutral/error tones and a **Start Chat** button (`ConnectionWizard.kt:1584`).
   *This card is the single best thing in the connect flow* — it tells the user exactly what
   works before they leave setup.
7. **Cold start into Chat** runs a narrated "startup sphere" checklist: state restored → route →
   hermes online → conversation ready (`RelayApp.kt:893-1011`), 650 ms floor, 12 s backstop.

### 3.2 Friction map of the journey

- **Tier labels on page 1.** The Welcome screen renders cards literally labeled "Standard" and
  "Advanced" (`OnboardingScreen.kt:317-324`) — the *first thing a new user reads* frames the
  product as a two-tier system, contradicting the project's own rule (`docs/plans/2026-06-07-
  standard-dashboard-relay-power-mode.md:20,426`: "Do not use visible Easy/Standard/Advanced
  tier language in the app UI").
- **Cold-start variance.** A code comment admits "6–28s launch variance against the same LAN
  server" (`RelayApp.kt:988`). The narrated checklist is a *clever* way to make a wait
  watchable, but it's still a wait — and Hermex's whole pitch is that it *just opens*.
- **No crisp setup promise.** Hermex says "three moves, ~15 minutes." Hermes-Relay's onboarding
  never sets a time/effort expectation, and the relay concept makes "how long will this take?"
  genuinely ambiguous to the user.

## 4. Standard vs Relay — the clarity audit

This is the conceptual heart of the product and the user's biggest mental-model risk.

**What's done well**
- Onboarding ordering, the primary "Standard Hermes" tile, and copy all lead with the
  no-plugin path and frame relay as optional (`ConnectionWizard.kt:995`, `OnboardingScreen.kt:471`).
- The readiness card's per-surface breakdown implicitly teaches the split (Chat/Manage/Voice
  work; Remote/Relay need pairing).
- Standard-vs-relay *degradation* is messaged consistently in Voice settings ("your server's
  configured TTS" when relay is absent, `VoiceSettingsScreen.kt:676`).

**Where it breaks**
- **The gate dead-loop (HIGH).** A Standard-only user who taps Terminal/Bridge/Relay-sessions
  hits `PowerFeatureGate` → "Requires pairing / Pair to unlock" (`PowerFeatureGate.kt:40`). It
  *never says the server-side prerequisite*: the **Relay plugin must be installed and running
  on the Hermes server**. A user who never installed the plugin taps "Pair to unlock," lands on
  the pair screen, and has nothing to pair to — a dead loop with no explanation. The
  `Unavailable` copy ("This server does not currently expose the relay capability") is still
  too abstract.
- **Three "code" concepts, no disambiguation (MEDIUM).** Standard *API key*, relay *6-char
  pairing code you type*, and *phone-generated code you give the host* (`ConnectionWizard.kt:
  1039`). "Pair Relay by code" vs "Show Relay code" is a subtle which-way-does-the-code-flow
  distinction. Direction-explicit copy ("Enter a code from the host" / "Give the host a code
  from this phone") would fix it.
- **Bridge route-gate contradicts its own screen (CRITICAL, see §6).**
- **Tier labels (see §3.2).**

**Benchmark note:** Hermex sidesteps *all* of this by having no relay — phone talks directly to
the server, one password, done. Hermes-Relay can't drop the relay (it's how terminal/bridge/
voice power works), so the answer is to make the relay **invisible until wanted**: Standard
should feel complete on its own, and relay-gated surfaces should explain the *server install*,
not just "pair."

## 5. Daily-use surfaces (Chat / Voice / Terminal)

### 5.1 Chat — **A−**

**Strengths:** Streaming is carefully de-jittered (auto-scroll distinguishes same-turn growth /
list-rebuild / new-bubble; reading history isn't yanked — `ChatScreen.kt:985`). Cold-start and
connecting states are *delightful* (skeleton bubbles + faux command-execution panel,
`ChatScreen.kt:2474`). **Tool transparency is best-in-class** — `ToolProgressCard` shows
preparing/running/completed/failed with duration, auto-collapse, and subagent guide-rail lanes;
this is what makes Hermes *feel* like it's working and builds real trust. Steer/Queue with
honest captions ("sends now — Hermes adjusts mid-turn" vs "delivered after this turn finishes")
is a standout.

**Top issues:**
- **HIGH — Slash commands give no in-place feedback.** `/new`, `/model`, `/compress` dispatch
  as a system-notice bubble that scrolls by; no toast, no haptic, no spinner on the command.
  On non-gateway routes a `/command` silently becomes a "slash commands available on gateway
  route" notice. Users won't trust `/compress` did anything. *Fix:* transient "Ran /compress ✓"
  chip + distinct haptic.
- **HIGH — Ambiguous pre-first-token gap.** On SSE/sessions paths the ~0.5–1.5 s before first
  token shows only animated dots; silence doesn't distinguish "connecting" from "thinking" from
  "slow model," and there's no escalation if the first token never lands. *Fix:* after ~4 s,
  escalate to a labeled "Still working…" state.
- **HIGH — Edit & resend is gateway-only and fails opaquely.** The Edit affordance only appears
  on gateway transport (`ChatScreen.kt:1789`); on SSE/sessions it's silently absent. When it
  *does* fail (conversation too long), the snackbar conflates "too long" with "wait for the
  turn." *Fix:* distinguish reasons; one-line hint that editing needs the gateway connection.
- **MEDIUM — Suggestion chips prefill instead of send** (`ChatScreen.kt:1675`) — a casual user
  expects a tap to start the conversation; requiring a second Send tap is friction at the exact
  zero-friction moment.
- **MEDIUM — Streaming vs final markdown "settle."** Lists/tables render as raw `- item` while
  streaming, then snap to formatted at completion (`MessageBubble.kt:321`). Stable but visibly
  jarring on every list reply.
- **LOW — Context meter only appears on gateway** (`ContextMeterBar.kt:36`); SSE/sessions users
  get no context warning before a sudden compression/failure.

### 5.2 Voice — **B**

**Strengths:** Full-screen overlay (MorphingSphere → waveform → transcript → state-morphing mic)
is genuinely delightful. Error recovery is thoughtful — the Retry clears *and* re-taps the mic
(`VoiceModeOverlay.kt:442`); the Stop-on-Speaking makes barge-in discoverable. Standard-vs-relay
voice degradation is messaged well.

**Top issues:**
- **HIGH — The voice entry point vanishes the moment you type.** The trailing slot is Send *or*
  Voice, never both (`ChatInputBar.kt:340`); type one word and the mic is gone, and the one-shot
  "Live voice conversation" hint only fires while the slot is VOICE — so a type-first user may
  never discover voice exists. *Fix:* persistent small mic in the left cluster next to "+".
- **MEDIUM — Voice settings is a 2,400-line single scroll** exposing raw "Provider ID / Model ID
  / Sample rate" fields to everyone (`VoiceSettingsScreen.kt:880`). *Fix:* lead with Engine +
  mode + one "Test voice"; collapse provider editing behind Advanced.
- **MEDIUM — "Auto-TTS" ships as a live toggle labeled "(coming soon)"** that does nothing
  (`VoiceSettingsScreen.kt:659`) — a trust leak in a 1.0. *Fix:* disable (grey) or remove.

### 5.3 Terminal — **B−**

**Strengths:** Real tmux session, Chrome-style tabs, always-visible extra-keys toolbar, slide-in
search, and the **Detach-vs-Kill education on tab close** (`TerminalScreen.kt:438`) is exactly
right for a tmux-backed session a non-expert would otherwise destroy.

**Top issues:**
- **HIGH — No auto-keyboard / no "tap to type" cue.** On attach nothing requests focus or shows
  the IME; the user stares at a live prompt with no keyboard and no affordance
  (`TerminalScreen.kt`, `TerminalWebView.kt`). *Fix:* request focus + show keyboard on attach,
  or an explicit first-attach hint.
- **HIGH — No copy/paste affordance.** Selecting text in xterm.js-in-WebView is notoriously hard
  on a phone, and there's no Paste button for the daily "paste this long command" action. *Fix:*
  Paste (clipboard → `sendInput`) + Copy-buffer in the extra-keys toolbar.
- **LOW — Tab labels are bare numbers** (`TerminalTabBar.kt:141`); four `hermes`/`htop`/`logs`/
  `build` tabs are indistinguishable. Auto-derive from the running command.

## 6. Hermes management (Manage / profiles / sessions) — **B**

**"Management" is split across four disjoint surfaces with three auth models** — itself a
finding:

| Surface | Transport / Auth | Read/Write |
|---|---|---|
| **Manage tab** (`DashboardManagementScreen.kt`) | Dashboard cookie | Read + write |
| **Profile Inspector** (`ProfileInspectorScreen.kt`) | Relay bearer | Read + SOUL/Memory write |
| **Chat agent sheet** (`AgentInfoSheet`) | Local `selectProfile()` | Per-conversation pick |
| **Analytics** (`AnalyticsScreen.kt`) | Local app stats | Read-only, on-device |

**Capability matrix vs the desktop dashboard** (condensed):

| Feature | View | Edit | Note |
|---|:--:|:--:|---|
| Profiles list / activate / create / delete | ✅ | ✅ | Full CRUD, `default` protected |
| Profile SOUL / description / model | ✅ | ✅ | Full-file SOUL editor |
| Profile **memory** | ✅* | ✅* | *Inspector only — not in Manage* |
| Skills list / toggle / hub search-preview-install | ✅ | ✅ | Strong |
| MCP list / toggle / test / catalog install | ✅ | ✅ | — |
| MCP **add custom server** | ❌ | ❌ | Missing |
| MCP **credentialed catalog install** | ⚠️ | ❌ | Install suppressed when env required (`:3108`) |
| Cron list / pause / trigger / delete / runs | ✅ | ✅ | — |
| Cron **create / edit a job** | ❌ | ❌ | **No create UI at all** |
| Models view / set main | ✅ | ✅ | Expensive-model confirm |
| Keys/env list / set / reveal / clear | ✅ | ✅ | — |
| **Config** | ⚠️ | ❌ | Schema *types* dump only — no values, no editing, no YAML |
| Provider OAuth / usage analytics / logs | ❌ | ❌ | Missing entirely |
| Sessions list / rename / delete | ✅ | ✅ | `SessionDrawer.kt` |

**Strengths:** Profiles/Skills/Keys/Models are *real read-write control*, not a viewer. The
caching engine (process cache + disk hydration + stale-while-revalidate + sibling pre-warm +
app-start pre-warm) makes re-entry instant and cold start land on cached data. Auth states are
handled with unusual care (word-based KPI strip, per-host cookie explainer, OAuth via in-app
WebView). Optimistic toggles + destructive confirms.

**Top issues:**
- **CRITICAL — Config tab is a dead-end dump** (`DashboardManagementScreen.kt:137,3001`). Tile
  promises "Schema and runtime options" but a user who taps in to *change* a setting can do
  nothing. *Fix:* wire `GET/PUT /api/config` (+`/config/raw`) or honestly relabel + show values.
- **CRITICAL — No way to create a Cron job** (`DashboardApiClient.kt:326`). "Automations" is a
  headline overview tile, yet a user with zero jobs lands on an empty list with no "+ New."
  Scheduling work from the phone is *the* daily-value use case that's missing.
- **HIGH — Credentialed MCP entries silently uninstallable** (`:3107`) — card shows "N
  credentials required" with no action and no path to supply creds. *Fix:* collect required env
  vars at install.
- **HIGH — Two parallel profile editors with no cross-link** (Manage cookie-auth vs Inspector
  relay-bearer; only Inspector has Memory, only Manage has create/delete/activate). A user
  editing a profile won't know the other half exists. *Fix:* unify, or deep-link + fold Memory
  into Manage so it works without relay.
- **HIGH — "Analytics" means two things; server usage/cost is absent.** The Settings→Analytics
  screen is on-device TTFT/token stats; the desktop's `/api/analytics/usage` (real spend) is
  nowhere. A self-hoster's daily "what did I spend?" has no answer. *Fix:* add a Usage section to
  Manage; rename the local screen "Diagnostics."
- **HIGH — Generic JSON summarization leaks raw shapes** ("5 fields", "Fields: providers,
  display…") when a payload isn't special-cased (`:3040,3269`), degrading to a debug console.

**Daily-use verdict:** Manage is *more than a rarely-touched admin panel* — switching the active
agent, flipping a skill, setting a model, adding a key, triggering an automation are plausible
daily actions, and the caching makes them feel instant. But it reads as a **strong control panel
for steady-state ops, not a creation surface**: you can drive what exists, but can't author
config or automations, which caps how "in control" a power user feels.

## 7. Bridge / Notifications / Settings / Errors

### 7.1 Bridge — **B+**

**This is above-average permission/safety UX.** The "why each permission" story is consistently
present (master-toggle info dialog, tiered checklist, "same perm as Wear OS/Android Auto/Tasker"
reassurance). Silent-deny is *engineered out* — every disabled tap opens the right Settings page
or fires an explanatory snackbar. Unattended access has layered, redundant visibility (one-time
scary dialog + inline keyguard alert + global amber banner + overlay chip) and is *honest* about
the lock-screen limitation. Destructive-verb confirmation shows the literal action + payload
("about to tap a button containing 'Send' — 'Send $500 to Alice'") and fails closed. Safety
defaults ship sensibly (banking apps + password managers pre-blocklisted).

**Top issues:**
- **CRITICAL — Route-gate contradicts the screen's own pre-pair copy.** `RelayApp.kt:1467` walls
  the entire Bridge screen behind `AuthState.Paired`, but `BridgeScreen.kt:262` has explicit
  "you can still configure permissions and safety before pairing" staged copy that is therefore
  **dead** — the most onboarding-friendly thing the team built (grant Accessibility + set safety
  *before* committing to pairing) is unreachable. *Fix:* let Bridge render in staged mode
  pre-pairing (the screen already handles it gracefully; the master toggle won't fire commands).
- **HIGH — The Accessibility grant is the scariest moment and the hand-off is blind.** The app
  preps the user but doesn't preview Android's full-screen "view and control your screen"
  warning. Cautious users abandon here. *Fix:* one-time interstitial previewing Android's exact
  words + reassurance (activity log, off-any-time).
- **HIGH — Destructive "Allow" button is solid red** (`DestructiveVerbConfirmDialog.kt:190`) —
  red is universal "stop"; putting it on *Allow* inverts muscle memory and risks mis-taps.
  *Fix:* make **Deny** the dominant safe button; Allow lower-emphasis (weighty via wording).
- **HIGH — PowerFeatureGate never names the server-side relay-plugin prerequisite** (the §4 dead
  loop).

### 7.2 Notifications — **B**

Honest framing and a clean grant flow, but the screen orders About → Status → Test
(`NotificationCompanionSettingsScreen.kt:122`), burying the action under explainer text. *Fix:*
Status + grant button first, About collapsible below.

### 7.3 Settings IA — **A−**

Deliberately de-mega-listed into Connections / Hermes / Power tools / App categories with a
single shared "plugin state" badge instead of badge noise (`SettingsScreen.kt:82,219`). This is
the right structure. Residual jargon leaks into subtitles ("foreground service indicator,"
"Plugin required/offline/stale," `tap_text / type … word-boundary match`).

### 7.4 Errors / empty states — **B**

`RelayErrorClassifier` is action-oriented — most `HumanError`s carry an `actionLabel` ("Re-pair,"
"Settings," "Retry," "Voice settings") and distinguish API-key-rejected vs session-expired vs
cert-mismatch, saying "server" not "relay" where appropriate. But the generic fallthroughs leak
raw exception text (`:171,181,194` — `t.message`, e.g. "failed to connect to /10.0.0.5:8767 …
after 10000ms"), and the Bridge activity log ships a developer-facing `Screenshot token:
hermes-relay://…` placeholder (`BridgeActivityLog.kt:201`). *Fix:* cap generic fallthrough to a
clean sentence (raw → Diagnostics); hide the token line.

## 8. Cross-cutting issues (the highest-leverage layer)

### CRITICAL
- **C1 — Expired/revoked pairing dead-ends with no recovery.** No distinct "Pairing expired →
  Pair again" state; the phone retries forever against a dead token showing only "Relay
  unreachable - tap to reconnect" (`RelayUiState.kt:163`). Because **the relay wipes in-memory
  sessions on every restart** (CLAUDE.md), this is the *single highest-frequency* failure and it
  currently looks like a hang. The plan doc explicitly specifies this state
  (`docs/plans/2026-06-07-…:111-116,294`); it's unimplemented. The `EmptyState`/`PairedDevices`
  CTA pattern already exists to copy.
- **C2 — Bridge route-gate vs staged-copy contradiction** (§7.1).
- **C3 — Manage can't *create* config or cron** (§6).

### HIGH
- **H1 — Context-blind Verify/timeout errors.** WSS-paired-but-API-401, host-hasn't-registered-
  the-code, and relay-down all surface the same generic "Timed out waiting for the relay…"
  (`ConnectionWizard.kt:322`), pointing at the wrong layer. `docs/remote-access.md:300` documents
  the API-401 case as literally looking like "pair succeeded, then silently dropped." *Fix:*
  tailor per cause.
- **H2 — Silent success/failure feedback loop** across slash commands, edit-resend, cancel/stop,
  voice unavailability (haptics + transient acknowledgments).
- **H3 — PowerFeatureGate omits the "install the Relay plugin on your server" prerequisite** (§4).
- **H4 — Voice entry vanishes on type; Terminal has no auto-keyboard/copy-paste** (§5).
- **H5 — Two profile editors / "Analytics" naming collision / credentialed-MCP wall** (§6).

### MEDIUM
- TTL picker asks a non-technical user a security-lifetime question mid-pairing
  (`ConnectionWizard.kt:2310`) — collapse to "Stay paired: 30 days (change)."
- Camera-permission denial on Scan is a silent dead-end (toast that vanishes) — fall through to
  manual URL entry.
- Inconsistent vocabulary: "Relay sessions" vs "Connections" vs "Paired devices" for one concept.
- Suggestion chips prefill instead of send; streaming-markdown settle; thinking-block 50-line cap.

## 10. Benchmark — vs Hermex (iOS) and the Hermes desktop dashboard

### 10.1 vs Hermex
**What Hermex does that we should steal:**
- **A crisp setup promise** — "Three moves: run, expose, connect. ~15 minutes." We set no
  expectation. Add a one-line "About X minutes" + a 3-step mental model to onboarding.
- **A workspace/file browser** — Hermex lets you "browse your workspace … the server's file
  system from the root." Hermes-Relay has **no file browser** at all. High-value, server-side
  data already exists via tools.
- **Model picker above the composer** with recent/favorites. We bury model switching in Manage +
  the agent sheet. Surfacing it one tap from chat is a daily-use win.
- **Home-screen shortcuts** (Memory / Insights / Projects) — fast lanes to common destinations.
- **An explicit offline read story** — "7 days / 5,000 messages" cache, advertised. We *have*
  Manage disk cache + chat cold-start cache but never promise offline reading of history.
- **Consumer-trust messaging** — "no relay or third-party service," "no analytics," "doesn't
  store your password after login." We should lead with our equivalent privacy story (everything
  is your server; relay is *your* server too).

**What we do that Hermex can't:** voice (full-screen agent + barge-in + provider-native
realtime), a real **terminal** (tmux), **device control / Bridge**, notification companion,
multi-endpoint LAN↔Tailscale↔public failover, and a far richer management surface. *This is the
moat.* The strategic move is to **neutralize Hermex's simplicity advantage on the Standard path**
so our power tools are pure upside.

> ⚠️ Naming note: "no relay or third-party service" is Hermex's *headline privacy claim*. Our
> "Relay" is **self-hosted on the user's own server**, but the word collides with exactly the
> thing Hermex tells users to avoid. Worth considering whether "Relay" is the right
> public-facing noun, or whether it should be framed as "the Hermes-Relay add-on / power pack."

### 10.2 vs the Hermes desktop dashboard
Manage translates the dashboard's *steady-state* controls well (profiles, skills, MCP manage,
models, keys) — arguably *better* than a raw web view on a small screen (optimistic toggles,
expensive-model guard, per-host auth). A desktop user hits walls on **Config editing**, **cron
creation**, **custom/credentialed MCP**, **provider OAuth**, **usage analytics**, and **logs**
(§6). Closing config-edit + cron-create would move Manage from "control panel" to "I can run my
whole agent from my phone."

## 11. "Painting Hermes in a good light" — verdict

**Does the app make agent use feel good, or frustrating?** Mostly *good*, with sharp edges.

**It builds confidence when:** tool calls stream as live, auto-collapsing cards; subagents get
guide-rail lanes; cold-start narrates real work instead of spinning; steer/queue give honest
captions; safety confirmations show the literal action. A user watching Hermes work through a
